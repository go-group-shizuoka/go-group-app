# 予約表OCR取り込み 設計書

最終更新: 2026-07-24
対象: GO GROUP APP（go-group-app / Vite+React / Supabase jjouwts）
目的: 保護者が毎月手書きする紙の利用予約表を、スタッフがスマホで1枚撮影するだけで
AIが読み取り→確認→GO GROUP APPの利用予定・送迎予定へ一括登録する。

## 1. 対象フォーム分析（実物 = 野々村様 / GO TOWN 1st/2nd / 令和8年8月）

### 1.1 レイアウト
| 位置 | 内容 |
|---|---|
| ヘッダ | `GO TOWN 1st/2nd　令和8年 8月　予約　名前 野々村 佳澄 さん` |
| 列構成 | 行番号(=日付1〜31) / 曜日 / 利用時間(開始〜終了) / 送迎(「迎」「送」印字) / 備考 |
| フッタ | `7月24日(金)締切りです…` |

### 1.2 記入ルール（現場の実態）
- **日付**: 一番左の行番号がそのまま日付（1〜31）。曜日は隣列に印字。
- **利用時間**: 来所する日だけ手書きで `9:00 〜 17:00` 等を記入。
- **送迎**: 印字済みの「迎」「送」の**文字を手書きで○で囲む**＝その送迎あり。囲みなし＝なし。
- **非利用日**: 利用時間が空欄、または**斜線**が引かれ、迎/送も未囲み。
- **休み**: 備考欄に「休み」と手書き（例: 8日・22日）＝欠席。
- **備考/イベント**: 「プール」「マックに行こう」「ビンゴ大会」「大社お祭り①②」「避難訓練」「誕生会」等。
  - ⚠ 重要: 備考のイベントは**来所日にも非利用日にも書かれる**（施設行事のため）。
    「その子が来る日」かどうかは**利用時間の記入と迎/送の○囲みで判定**する。備考の有無では判定しない。

### 1.3 実運用の画像条件
- 手書き（時間・○囲み・イベント名）
- 斜め撮影・用紙のたわみ・照明ムラ・影
- 上部に手書きメモ（`8/20 ありがとうございます`）等のノイズが乗ることがある

## 2. 読み取り仕様（抽出項目と判定ルール）

### 2.1 シート単位
| 項目 | 抽出元 | 変換 |
|---|---|---|
| 対象年月 | ヘッダ「令和8年 8月」 | 和暦→西暦（令和8年=2026）→ `year=2026, month=8` |
| 利用者名 | ヘッダ「名前 …さん」 | `野々村 佳澄`（利用者マスタと照合） |
| 施設名 | ヘッダ「GO TOWN 1st/2nd」 | 参考表示。登録先施設はスタッフが選択（f3/f4） |

### 2.2 行（日）単位の判定ロジック ★本フォーム最重要
各行（日付1〜31）について:

```
IF 利用時間に時刻が記入されている（斜線でない）:
    status = "来所"
    startTime / endTime = 記入時刻
    pickup  = 「迎」が○で囲まれていれば true
    dropoff = 「送」が○で囲まれていれば true
    memo    = 備考テキスト（イベント名含む）
ELSE IF 備考に「休み」「欠」等がある:
    status = "欠席"（送迎・時刻は無し）
ELSE:
    その日は出力しない（予約なし。斜線＋イベントのみの行も含む）
```

- **来所** → `planned_visits` へ（利用予定・送迎予定）
- **欠席** → `att_data`（出欠）へ status=欠席
- **非利用（斜線/空欄）** → 登録しない（備考のイベントは将来「イベント表」で別途取り込み）

## 4. AI構造化プロンプト（本フォーム向け改訂案）

現行 `yotei` プロンプトに対する改訂ポイント:
1. 日付は「行番号＝日付」であることを明示（○を日付上に探さない）。
2. 送迎は「『迎』『送』の**文字が○で囲まれているか**」で判定（別の○印ではない）。
3. 斜線・空欄の非利用日は**出力しない**。備考のイベントの有無で来所判定しない。
4. 「休み」「欠」→ status="欠席"。
5. 手書き・かすれは confidence を正直に下げる。

```text
この障害児通所サービスの月間利用予定表（手書き）を解析してください。
和暦は西暦に変換（令和8年=2026）。表は「行番号＝その月の日付(1〜31)」です。

各行を次で判定してください：
- 利用時間(例 9:00〜17:00)が手書きで記入されている → status:"来所"
- 利用時間が空欄／斜線で、備考に「休み」「欠」等 → status:"欠席"
- 上記以外（空欄・斜線のみ、備考にイベント名だけ）→ その行は出力しない

送迎は印字された「迎」「送」の文字が手書きの○で囲まれているかで判定：
- 「迎」が○で囲まれている → pickup:true
- 「送」が○で囲まれている → dropoff:true
欠席日は pickup/dropoff=false、startTime/endTime=null。

JSONのみ返す：
{
  "childName":"利用者氏名", "nameConfidence":0-100,
  "facilityName":"施設名", "year":2026, "month":8,
  "visits":[{"date":3,"dayOfWeek":"月","status":"来所",
    "startTime":"09:00","endTime":"17:00","pickup":true,"dropoff":true,
    "memo":"備考(イベント名含む。なければnull)","event":"イベント名(なければnull)",
    "confidence":0-100}]
}
上部の手書きメモや締切注記など表外の文字は無視。confidenceは判読しにくさを反映。
```

> 実装は `api/ocr.js` の `prompts.yotei` を差し替えるだけ（他モード・他機能に影響なし）。

## 5. 画像前処理・斜め補正・手書き認識の方針

### 5.1 基本方針（過剰処理をしない）
Claude Opus 4.5 Vision は**傾き・照明ムラ・手書き・○囲みに元々強い**。
まずは素の画像＋プロンプトで精度を測り、不足時のみ前処理を足す（保守性優先）。

### 5.2 段階的な前処理（必要に応じて）
| レベル | 処理 | 実装 | 目的 |
|---|---|---|---|
| L0(既定) | 長辺〜2000pxへ縮小、JPEG品質85 | Canvas | 転送/コスト削減・十分な解像度確保 |
| L1 | グレースケール＋自動コントラスト | Canvas filter | かすれ手書きの視認性 |
| L2 | 台形補正（deskew/perspective） | jscanify or OpenCV.js（動的import） | 斜め撮影の矯正 |

- L2は重い（OpenCV.js ~8MB）。**初期は導入せず**、撮影UIで「平置き・正対・影なし」を案内。
- 精度が実測で不足した場合に限りL2をPhaseで追加（遅延ロードでバンドル影響を局所化）。

### 5.3 撮影UIガイド（低コストで効果大）
- 「用紙を平らに置き、影が入らないよう真上から撮影」ガイド表示（既存の使い方カードに追記）。
- 端が切れないよう枠ガイド（任意）。

## 6. Supabase 保存設計（既存テーブル再利用・新規作成しない）

| 種別 | テーブル | 主な列 | 備考 |
|---|---|---|---|
| 来所予定 | `planned_visits` | child_id, facility_id, visit_date, start_time, end_time, pickup_required, dropoff_required, memo, event_name | 予約詳細の本命。RLS(施設分離)済 |
| 送迎/カレンダー | `schedules`(+scheduleData) | user_id, date, status, transport_to, transport_from | 表示連動。時間列なし(§8課題) |
| 出欠 | `att_data` | user_id, att_date, status | 休み→"欠席"。RLS済 |
| 施設イベント(将来) | `facility_events` | facilityId, eventDate, title | 備考イベントの取り込み先(将来) |

- 冪等キー: planned_visits `id = child_日付`、att_data `id = child_日付` → 再取込時は上書き（既存更新時 created_at 保持）。
- **SaaS(他法人)展開時**は planned_visits/att_data/schedules に `org_id` を付与（SaaS用DBは対応済スキーマ）。GO GROUP単独運用は facility_id で十分。

## 7. 既存OCR機能との共通化（保守性・再利用性）

- **OCRエンジン**: `/api/ocr` の `mode` 方式に統一（jukyusha/soudan/isp_draft/monitoring_memo/**yotei**）。
  共通処理＝認証(requireUser)＋画像/PDF両対応＋JSON抽出。新種は**プロンプト追加のみ**。
- **確認画面**: `ScheduleOCRScreen`。将来の複数フォームは
  汎用 `OcrImportScreen(mode, schema, mapper)` へ一般化（確認テーブルを列定義から自動生成）。
- **運営AI**: `analyzeSchedule()`（チェック）＋共通表示 `AiReportCard`＋`buildScheduleReport()`。
  送迎表/学校予定表/イベント表/職員シフトは `build◯◯Report()` を足せば同カードを再利用。

## 8. 既知の課題・リスク

| 区分 | 内容 | 対応 |
|---|---|---|
| プロンプト | 現行yoteiは「○のある日/時間のある日」を来所とみなす汎用形。本フォームの
「行番号=日付」「迎送の○囲み」「斜線非利用」を精緻化要 | §4改訂(Phase1) |
| 精度 | 手書き・斜め・影で送迎○囲みの誤認識の可能性 | 確認画面で目視修正必須。信頼度<80%黄/<60%赤で注意喚起(実装済) |
| データ整合 | schedulesに start/end 列がなく取りこぼし | planned_visits一本化 or 列追加(Phase3) |
| 施設 | ヘッダ「1st/2nd」で施設が一意でない | 登録先はスタッフが選択(実装済) |
| 複数児童 | 1枚1児童前提（本フォームも1名） | 複数名シートは将来(Phase5) |
| コスト | Opus Vision呼出は高コスト | 認証ガード済。施設単位レート制限を検討 |
| イベント | 備考のイベント(プール/お祭り)は現状memo止まり | 将来「イベント表」へ構造化取り込み |

## 9. LINE入力対応（将来設計）

現行に無理なく拡張できる形にしておく:

```
保護者LINE で予定表を撮影送信
  → LINE Webhook(api/line-webhook.js 既存) が画像を受信
  → 同じ /api/ocr mode:"yotei" で解析（エンジン共通）
  → 確認は2案:
     (a) スタッフがアプリのScheduleOCRScreenで確認・登録（安全・推奨・初期）
     (b) 保護者がLIFF画面で確認・確定（将来・要権限設計）
```
- エンジン(/api/ocr)と登録ロジック(planned_visits/att_data)を共通化してあるため、
  LINE経路は「入口(画像取得元)」の差し替えだけで載る。
- 保護者確定(b)はRLS/なりすまし対策が要るため、まずはスタッフ確認(a)で運用開始。
